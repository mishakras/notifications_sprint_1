# Результаты сравнения производительности MongoDB vs PostgreSQL

## Методология тестирования

### Конфигурация тестового окружения:
- **Объем данных**: 100,000, 1,000,000, 5,000,000 и 10,000,000 записей
- **Целевое время отклика**: 200.0 мс
- **Количество итераций**: 3
- **Параллельные процессы**: 4
- **Размеры батчей**: 1,000, 5,000, 10,000, 25,000, 50,000, 100,000
- **Платформа**: Docker containers
- **Устройство**: MacBook M1 Pro

### Используемые скрипты:
- `mongo_test.py` - тестирование MongoDB
- `postgres_test.py` - тестирование PostgreSQL
- `perfomance_comparison.py` - запуск всех тестов и сбор метрик

### Структура данных:
- **Лайки**: 70% от общего объема (user_id, movie_id, rating 0-10, timestamp)
- **Рецензии**: 20% от общего объема (user_id, movie_id, title, content, rating, likes/dislikes, timestamp)
- **Закладки**: 10% от общего объема (user_id, movie_id, timestamp)

## Основные результаты тестирования

### 1. Производительность вставки данных

#### Малые объемы данных (100,000 записей)

**100,000 записей, batch 1,000:**
- MongoDB (Likes): 68,133 записей/сек
- PostgreSQL (Likes): 30,624 записей/сек
- **MongoDB быстрее в 2.22 раза**

**100,000 записей, batch 5,000:**
- MongoDB (Likes): 68,662 записей/сек
- PostgreSQL (Likes): 50,634 записей/сек
- **MongoDB быстрее в 1.36 раза**

#### Средние объемы данных (1,000,000 записей)

**1,000,000 записей, batch 10,000:**
- MongoDB (Likes): 47,119 записей/сек
- PostgreSQL (Likes): 53,566 записей/сек
- **PostgreSQL быстрее в 1.14 раза**

**1,000,000 записей, batch 25,000:**
- MongoDB (Likes): 50,477 записей/сек
- PostgreSQL (Likes): 55,848 записей/сек
- **PostgreSQL быстрее в 1.11 раза**

#### Большие объемы данных (5,000,000+ записей)

**5,000,000 записей, batch 50,000:**
- MongoDB (Likes): 31,268 записей/сек
- PostgreSQL (Likes): 39,683 записей/сек
- **PostgreSQL быстрее в 1.27 раза**

**10,000,000 записей, batch 100,000:**
- MongoDB (Likes): 16,314 записей/сек
- PostgreSQL (Likes): 30,519 записей/сек
- **PostgreSQL быстрее в 1.87 раза**

## Ключевые выводы

1.  **Производительность вставки**:
    *   На малых объемах данных MongoDB показывает лучшие результаты.
    *   На средних и больших объемах PostgreSQL демонстрирует превосходство.
    *   PostgreSQL лучше масштабируется при увеличении объема данных.

2.  **Производительность запросов**:
    *   На малых и средних объемах обе СУБД показывают сопоставимые результаты, с небольшим преимуществом MongoDB на 100K записях.
    *   На больших объемах PostgreSQL значительно опережает MongoDB без шардирования.
    *   При 10M записей разница в производительности запросов достигает 50 раз в пользу PostgreSQL.

3.  **Соответствие требованиям**:
    *   Обе СУБД удовлетворяют требованию по времени отклика (200 мс).
    *   PostgreSQL без масштабирования демонстрирует более стабильные результаты при увеличении нагрузки.

## Итоговое заключение
PostgreSQL продемонстрировал лучшую производительность при чтении данных, особенно на средних и больших объёмах (от 1 до 10 миллионов записей)
MongoDB показал высокую скорость вставки данных на малых объёмах (до 100–500 тысяч записей) и при небольших размерах батчей, что может быть полезно в сценариях с частыми обновлениями.
Кроме того, MongoDB обладает встроенной возможностью горизонтального масштабирования через шардирование, что даёт ей преимущество при работе в высоконагруженной среде с потенциальным ростом объёмов данных.

На моём устройстве PostgreSQL показал более высокую стабильность и скорость чтения, что делает его предпочтительным выбором в текущей конфигурации. Все тестовые запросы укладывались в требуемое время отклика (200 мс), но PostgreSQL уверенно опережал MongoDB на больших объемах.

Тем не менее, при переходе к масштабируемому окружению с более мощной инфраструктурой и необходимостью горизонтального масштабирования (например, при росте до 100+ млн записей) MongoDB может стать более подходящим вариантом за счёт гибкой архитектуры и шардирования.

Таким образом:

В текущих условиях — предпочтительнее PostgreSQL.
В перспективе масштабирования и увеличения нагрузки — MongoDB становится более конкурентоспособным вариантом.
